# Refactoring TODO List

This document outlines areas for improvement in the codebase, focusing on refactoring, optimization, and adherence to best practices.

## Code Duplication & DRY Violations

- **[MEDIUM] Duplicated Aggregate Functions:** The aggregate functions (`sumAggregate`, `countAggregate`, `avgAggregate`, `minAggregate`, `maxAggregate`) in `core/ephemeral/aggregates.go` are very similar in structure. They all iterate over records, extract a value, and perform a calculation. 
  - **Action:** Refactor these into a single generic function that takes the aggregation logic as a parameter. This will reduce code duplication and make it easier to add new aggregate functions.

- **[LOW] Repetitive `convertToFilterValue` Logic:** The `convertToFilterValue` method is duplicated in `query/builder.go` within `FilterConditionBuilder` and `FilterConditionBuilderInGroup`. 
  - **Action:** Create a single, shared `convertToFilterValue` utility function to be used by both builders.

## Optimization Opportunities

- **[LOW] Query Plan Caching:** The `QueryEngine` in `core/query/executor.go` has a cache for partitioned queries. However, the cache key is generated by marshaling the entire query DSL to JSON, which can be slow. 
  - **Action:** Consider a more efficient cache key generation strategy, such as a hashing function that operates directly on the query structure without JSON marshaling.

## Refactoring Candidates

- **[MEDIUM] Event Emission Logic:** The `withEventEmission` function in `core/persistence/events.go` is a good start, but the event creation is verbose. 
  - **Action:** Create a helper function or a builder for `PersistenceEvent` to simplify the creation of start, success, and failure events. This will make the `Create`, `Read`, `Update`, and `Delete` methods in `core/persistence/events.go` cleaner.

- **[MEDIUM] `QueryHelper` Complexity:** The `QueryHelper` in `core/query/helper.go` is becoming very large and is responsible for filtering, sorting, pagination, projection, and aggregation. 
  - **Action:** Break down the `QueryHelper` into smaller, more focused components. For example, have separate helpers for filtering, sorting, and aggregation. This will improve modularity and make the code easier to maintain.

## Misplaced Concerns & Poor Modularization

- **[HIGH] Persistence Logic in `ephemeral` Package:** The `ephemeral` package should ideally only contain the in-memory implementation of the `DatabaseInteractor`. However, `core/ephemeral/aggregates.go` contains business logic for aggregations. 
  - **Action:** Move the aggregation logic to the `query` package, as it is a query-level concern, not a storage-level one. The `EphemeralDatabaseInteractor` can then call these functions.

- **[MEDIUM] `parser` in `query`:** The `parser` package is located inside the `query` package. While related, a query parser is a distinct component. 
  - **Action:** Consider moving the `parser` package to its own top-level package (e.g., `core/parser`) to better separate concerns.

## Anti-Patterns & Non-Idiomatic Go

- **[LOW] Extensive Use of `any`:** The codebase uses `any` (or `interface{}`) in many places, particularly in the persistence and query layers. While sometimes necessary for a generic query engine, it sacrifices type safety. 
  - **Action:** Where possible, use generics to provide more type safety. For example, the `Create` method in `PersistenceCollectionInterface` could be generic.

- **[LOW] Pointer to Primitives:** The use of pointers to primitive types (e.g., `*string`, `*bool`) in structs like `PartialFieldDefinition` is not very idiomatic in Go. It's often better to use a separate boolean flag to indicate if a field was present. 
  - **Action:** Re-evaluate the use of pointers to primitives. Consider using a different approach, such as a map of fields to update, to avoid this pattern.
