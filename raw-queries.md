# Revised Proposal: Pluggable Raw Query Functionality

This document proposes the integration of a pluggable raw query execution capability into the `go-anansi` framework. This feature provides an essential "escape hatch" for developers to execute highly optimized, complex, or database-specific operations that may not be fully expressible or efficiently handled by the framework's Domain-Specific Language (DSL). It acknowledges the need for direct database interaction while maintaining a structured approach to ensure safety and clarity.

## 1. Current Architecture Analysis

The `go-anansi` framework is built on a robust architecture that abstracts database interactions through a generic `query.Query` DSL and a `NativeQueryBuilder` interface. The framework partitions queries into database-executable and in-memory "residual" parts, ensuring portability and consistency.

However, this abstraction, while powerful, can sometimes limit developers who need:
*   To leverage advanced, database-specific features not exposed by the DSL.
*   To write highly optimized queries that outperform those generated by the framework.
*   To perform DDL (Data Definition Language) or other database commands directly.

The proposed raw query functionality extends this architecture by providing a controlled mechanism for direct database access, complementing the existing DSL.

## 2. Proposed Raw Query Functionality

We propose a comprehensive raw query solution that allows developers to execute native database commands with templating capabilities for collection name resolution and structured options for database-specific context.

### Core Concepts

*   **Raw Query as an Escape Hatch:** This feature is intended for scenarios where the framework's DSL is insufficient, or direct optimization is required.
*   **Templating for Abstraction:** Physical database names (tables, collections) are abstracted using a templating mechanism, ensuring portability of raw queries across schema versions.
*   **Structured Options:** Database-specific parameters and commands are provided via a structured `Options` map, allowing for fine-grained control.
*   **Parameterized Arguments:** Arguments are passed separately to prevent injection vulnerabilities.
*   **Integration with Existing DSL:** Raw queries are integrated directly into the existing `query.Query` struct, with raw queries taking precedence when present.

### Raw Query Definition (`core/query/dsl.go`)

The core of this functionality is the `RawQuery` struct, which encapsulates all components of a raw database operation:

```go
// RawQueryTarget specifies a collection and its schema version
// to be used in a raw query.
type RawQueryTarget struct {
    Collection string `json:"collection"`
    Version    string `json:"version,omitempty"` // If omitted, resolves to the active version
}

// RawQuery represents a raw query payload for execution.
// It combines a string-based query template with structured options.
type RawQuery struct {
    // Template is a string-based query.
    // For SQL, this is the SQL string.
    // For MongoDB, this could be a JSON string representing the filter,
    // an aggregation pipeline, or a command body.
    // Placeholders like {{collection}} are resolved here using the 'From' map.
    Template string `json:"template,omitempty"`

    // Options provides structured, database-specific parameters that modify the query's behavior.
    // For SQL, this might be hints, expectRows boolean for schema inclusion.
    // For MongoDB, this could be limit, sort, projection, includeSchema, etc.
    // This is a map to allow flexible, database-specific key-value pairs.
    Options map[string]any `json:"options,omitempty"`

    // Maps placeholder names in the 'Template' string to specific collection targets.
    // This field is only relevant when 'Template' is used.
    From map[string]RawQueryTarget `json:"from,omitempty"`

    // Args are parameters for the query. Primarily for SQL.
    // For MongoDB, if Template is a JSON string, Args could map to custom placeholders
    // within that JSON string (e.g., {{arg0}}, {{arg1}}).
    Args []any `json:"args,omitempty"`
}
```

### Enhanced Query DSL Integration (`core/query/dsl.go`)

The existing `Query` struct is extended to include raw query capability:

```go
// Enhanced Query struct
type Query struct {
    // Existing DSL fields...
    Filter     *Filter     `json:"filter,omitempty"`
    Projection *Projection `json:"projection,omitempty"`
    Sort       *Sort       `json:"sort,omitempty"`
    Limit      *int        `json:"limit,omitempty"`
    
    // New raw query field - takes precedence over DSL fields when present
    Raw *RawQuery `json:"raw,omitempty"`
}
```

### Raw Query Result (`core/persistence/base/types.go`)

A new `RawQueryResult` struct is introduced to handle the diverse outcomes of raw database operations:

```go
// RawQueryResult represents the outcome of executing a raw database query.
// It provides a flexible container for various types of query results.
// Developers using raw queries are expected to understand their specific
// database and query type, and interpret results accordingly.
type RawQueryResult struct {
    // Data for SELECT queries. Can be a slice of data.Document or a more generic type.
    // The specific type depends on the database implementation and query type.
    Data        any `json:"data,omitempty"`
    
    // Count of rows returned for SELECT queries.
    Count       int `json:"count,omitempty"`

    // AffectedRows for INSERT, UPDATE, DELETE queries.
    AffectedRows int64 `json:"affectedRows,omitempty"`

    // Message provides additional status or information for any query type.
    Message     string `json:"message,omitempty"`
    
    // Success indicates if the query executed without error.
    Success     bool   `json:"success"`
}
```

### Entry Points

Raw queries can be executed via two primary entry points:

1.  **Collection-Scoped Raw Queries:** For queries logically tied to a single collection.
    *   **User Experience:** The user constructs a `query.Query` object with the `Raw` field populated and passes it to the existing `Collection.Read()` method.
    *   **Implementation:** The `NativeQueryBuilder` will detect the `Raw` field in the `query.Query` object and delegate its processing accordingly. When `Raw` is present, it takes precedence over DSL fields.

2.  **Global/Unscoped Raw Queries:** For queries that operate across multiple collections, query system tables, or perform database-level operations not tied to a single collection.
    *   **User Experience:** A new `Query()` method will be available on the `BasePersistence` interface.
    *   **Implementation:** `BasePersistence.Query()` will create a `query.Query` with the provided `RawQuery` and delegate to the `NativeQueryBuilder`.

## 3. Architectural Components

The new functionality will be implemented through extensions to existing interfaces and components.

### 1. Enhanced `BasePersistence` Interface (`core/persistence/base/types.go`)

The `BasePersistence` interface will gain a new method for executing global raw queries:

```go
// Enhanced BasePersistence interface
type BasePersistence interface {
    // ... existing methods ...

    // Query executes a raw, templated query directly against the database.
    // This allows for operations that are not tied to a specific collection,
    // or for highly optimized, custom queries.
    Query(ctx context.Context, rawQuery *RawQuery) (*RawQueryResult, error)
}
```

### 2. `NativeQueryBuilder` Integration

The existing `NativeQueryBuilder` interface handles raw query processing through its `Build` method:

```go
// Existing NativeQueryBuilder interface (no changes required)
type NativeQueryBuilder[T any] interface {
    Build(q *query.Query, stmtType StatementType, extra any) (Query[T], error)
}
```

Each database-specific implementation will be enhanced to detect and process raw queries:

```go
// Example: SQLite implementation
func (s *SQLiteQueryBuilder) Build(q *query.Query, stmtType StatementType, extra any) (Query[string], error) {
    // Check for raw query first - raw takes precedence
    if q.Raw != nil {
        return s.buildRawQuery(q.Raw, stmtType)
    }
    
    // Fall back to existing DSL processing
    return s.buildDSLQuery(q, stmtType, extra)
}

func (s *SQLiteQueryBuilder) buildRawQuery(raw *RawQuery, stmtType StatementType) (Query[string], error) {
    // 1. Resolve collection names from raw.From using collectionRegistry
    resolvedTemplate, err := s.resolveCollectionPlaceholders(raw.Template, raw.From)
    if err != nil {
        return nil, err
    }
    
    // 2. Process argument placeholders in template
    finalQuery, err := s.processArgumentPlaceholders(resolvedTemplate, raw.Args)
    if err != nil {
        return nil, err
    }
    
    // 3. Handle database-specific options
    var schema *schema.SchemaDefinition
    if expectRows, ok := raw.Options["expectRows"].(bool); ok && expectRows {
        // Include schema for result mapping when rows are expected
        schema = s.determineResultSchema(raw.From)
    }
    
    // 4. Create native query
    return &NativeQuery[string]{
        Query:  &sqlQuery{statement: finalQuery, stmtType: stmtType},
        Schema: schema,
    }, nil
}
```

### 3. Database-Specific Option Handling

Each `NativeQueryBuilder` implementation will define and handle its own set of options:

**SQLite Options:**
```go
// SQLite-specific options
raw := &RawQuery{
    Template: "SELECT * FROM {{users}} WHERE active = ?",
    Options: map[string]any{
        "expectRows": true,           // Include result schema
        "timeout":    "30s",          // Query timeout
        "pragma":     "journal_mode", // SQLite pragma settings
    },
}
```

**MongoDB Options:**
```go
// MongoDB-specific options
raw := &RawQuery{
    Template: `{"find": "{{users}}", "filter": {"active": true}}`,
    Options: map[string]any{
        "limit":          100,
        "sort":           bson.M{"_id": -1},
        "readPreference": "secondary",
        "includeSchema":  true,
    },
}
```

### 4. Template Processing Engine

Each `NativeQueryBuilder` implementation will include a template processor capable of:

1. **Collection Name Resolution**: Replacing `{{collection}}` placeholders with physical table/collection names
2. **Argument Substitution**: Processing parameterized arguments according to database conventions
3. **Validation**: Ensuring all placeholders are properly resolved

## 4. Usage Examples

### Collection-Scoped Raw Query

```go
// Using existing Collection.Read() method with raw query
rawQuery := &query.Query{
    Raw: &query.RawQuery{
        Template: "SELECT id, name, created_at FROM {{users}} WHERE created_at > ? ORDER BY created_at DESC",
        From: map[string]query.RawQueryTarget{
            "users": {Collection: "users", Version: "v1"},
        },
        Args: []any{time.Now().AddDate(0, -1, 0)},
        Options: map[string]any{
            "expectRows": true,
            "timeout":    "30s",
        },
    },
}

result, err := collection.Read(ctx, rawQuery)
```

### Global Raw Query

```go
// Using new BasePersistence.Query() method
rawQuery := &query.RawQuery{
    Template: "CREATE INDEX idx_user_email ON {{users}} (email)",
    From: map[string]query.RawQueryTarget{
        "users": {Collection: "users", Version: "v1"},
    },
    Options: map[string]any{
        "expectRows": false,
    },
}

result, err := persistence.Query(ctx, rawQuery)
if result.Success {
    log.Printf("Index created: %s", result.Message)
}
```

### Cross-Collection Query

```go
// Query spanning multiple collections
rawQuery := &query.RawQuery{
    Template: `
        SELECT u.name, p.title 
        FROM {{users}} u 
        JOIN {{posts}} p ON u.id = p.author_id 
        WHERE u.active = ? AND p.published_at > ?`,
    From: map[string]query.RawQueryTarget{
        "users": {Collection: "users", Version: "v1"},
        "posts": {Collection: "posts", Version: "v2"},
    },
    Args: []any{true, time.Now().AddDate(0, -1, 0)},
    Options: map[string]any{
        "expectRows": true,
    },
}

result, err := persistence.Query(ctx, rawQuery)
```

## 5. Benefits and Trade-offs

### Benefits

*   **Unrestricted Power:** Provides an "escape hatch" to leverage the full capabilities of the underlying database, including advanced features and DDL operations.
*   **Optimization:** Allows for hand-tuned queries that can achieve superior performance for critical paths.
*   **Flexibility:** Supports diverse database paradigms (SQL, NoSQL) through a unified, yet adaptable, `RawQuery` structure.
*   **Abstraction of Physical Names:** Templating ensures that raw queries remain portable across schema versions by abstracting physical table/collection names.
*   **Security:** `Args` field promotes parameterized queries, mitigating SQL/NoSQL injection risks.
*   **Seamless Integration:** Raw queries integrate directly into existing query workflows with precedence-based handling.
*   **Database-Specific Optimization:** Each `NativeQueryBuilder` implementation can optimize raw query processing for its specific database.

### Trade-offs

*   **Reduced Portability:** Raw queries are inherently tied to the specific database language and features, making applications less portable across different database backends.
*   **Loss of Framework Features:** Raw queries do not engage the framework's higher-level features such as:
    *   **Residual Query Processing:** In-memory filtering, sorting, and other operations performed by the framework are not engaged.
    *   **Search Integration:** The "Search-as-a-Filter" functionality, which relies on inspecting the DSL, is not applicable to raw queries.
*   **Increased Developer Responsibility:** Developers using raw queries assume greater responsibility for query correctness, performance, and database compatibility.
*   **Implementation Complexity:** The `NativeQueryBuilder` implementations require sophisticated logic to parse, template, and combine the various fields of the `RawQuery` for each database type.

## 6. Implementation Plan

The implementation will be broken down into the following discrete steps:

### Phase 1: Core Infrastructure
1. **Extend `query.Query` struct** to include the `Raw *RawQuery` field
2. **Define `RawQuery` and `RawQueryResult` types** with comprehensive documentation
3. **Add `Query()` method** to the `BasePersistence` interface

### Phase 2: Template Processing Engine
1. **Implement template resolution logic** for collection name placeholders
2. **Add argument substitution capabilities** for parameterized queries
3. **Create validation framework** for template and placeholder verification

### Phase 3: Database-Specific Implementations
1. **Enhance existing `NativeQueryBuilder` implementations** to detect and process raw queries
2. **Implement database-specific option handling** (expectRows for SQL, includeSchema for MongoDB, etc.)
3. **Add result mapping logic** based on database-specific options

### Phase 4: Testing and Documentation
1. **Comprehensive unit tests** for template processing and validation
2. **Integration tests** with real database backends
3. **Performance benchmarks** comparing raw queries to DSL equivalents
4. **Complete documentation** with usage examples and best practices

## 7. Testing Strategy

### Unit Tests
*   **Template Processing:** Test placeholder resolution, argument substitution, and edge cases
*   **Validation Logic:** Test malformed templates, missing collections, type mismatches
*   **Option Handling:** Test database-specific option parsing and validation

### Integration Tests
*   **Multi-Database Support:** Test raw queries across SQL and NoSQL backends
*   **Schema Integration:** Test result mapping with and without schema inclusion
*   **Error Scenarios:** Test network failures, syntax errors, and recovery

### Performance Tests
*   **Benchmark raw queries** against equivalent DSL queries
*   **Memory usage analysis** for large result sets
*   **Template processing overhead** measurement

## 8. Conclusion

This revised proposal for pluggable raw query functionality introduces a powerful and flexible mechanism for direct database interaction within the `go-anansi` framework. By integrating raw queries directly into the existing `query.Query` structure and leveraging the established `NativeQueryBuilder` architecture, it provides a controlled "escape hatch" that balances the need for direct database access with the framework's principles of abstraction and safety.

The design acknowledges the inherent trade-offs of raw queries (portability, loss of framework features) but provides a robust solution for scenarios where direct database control is paramount. By distributing implementation complexity to database-specific components and providing flexible option handling, this feature will empower developers to optimize critical paths and leverage database-specific capabilities while maintaining the framework's architectural integrity.

The seamless integration with existing workflows, combined with the precedence-based approach (raw queries take priority when present), ensures that this powerful feature enhances rather than disrupts the current development experience.
